# 数据收集策略

Collect 原语的 URL 抓取步骤的详细流程。适用于 `research.Collect` 需要检索 URL（或外部托管产物）并保存至 `raw/ext-<name>.md` 的所有场景。

> **调用来源**：[`../SKILL.md`](../SKILL.md)（Collect 原语，步骤 3）。
> **也被引用自**：`wiki` skill 的 source-* 摄取路径（跨 skill 复用）。

## 核心原则

1. **用户提供的 URL 是强制性的**。用户主动发送 URL 说明该来源重要，必须获取。穷尽工具阶梯，需要登录则暂停请求用户协助，不可静默跳过。
2. **按失败信号升级，不按预设分类选择**。先用最廉价的工具尝试，观察返回结果，按信号升级到更强工具。
3. **明确的失败记录**。穷尽所有选项后仍无法获取 → 记录完整失败路径，不要静默省略。

---

## 1. 工具选择 — 按信号升级

不要预先按 URL 类型分桶；从最廉价的工具开始，按返回信号决定是否升级。

- **起手**：GitHub 仓库/文件用 `deepwiki`；已知库/框架文档站用 `context7`；其他 URL 一律先用内置 `fetch`。
- **升级到 `playwright`**：fetch 返回空内容 / 残缺 HTML / 反爬挑战页面，或页面有标签切换、分页、"加载更多"、需要登录等动态状态。需登录时 playwright 暂停，请用户协助后继续。
- **发现 JSON API**：页面背后有可发现的干净端点时，直接 `fetch` 该端点（比抓渲染页快且稳）。
- **只有主题不知 URL**：先用 `exa`（语义）或 `grok-search`（时效）定位，再回到本流程。

工具速记：`deepwiki` 仓库语义问答 / `context7` 框架文档段 / `fetch` 静态 HTML·RSS·JSON / `playwright` 真实浏览器 / `exa` 语义搜 / `grok-search` 时效搜。

---

## 2. URL 分级 — 用户 URL vs 发现 URL

### 用户提供的 URL（强制获取）

用户主动粘贴或指明的 URL 是研究意图的一部分。规则：

- 穷尽工具阶梯，直到获取可用内容或穷尽所有选项
- 需要登录 → **暂停**，告知用户："此 URL 需要登录。请在浏览器中完成登录后告知我，我将继续抓取。" 等待用户确认后继续
- 穷尽所有工具仍失败 → 记录为 `raw/ext-<name>-failed.md`（见第 5 节）
- **永远不要**静默跳过用户提供的 URL

### 研究过程中发现的 URL（按相关性判断）

在收集过程中从页面内容、搜索结果等发现的 URL：

- 与意图锚定的 success criteria 相关 → 追踪，按本流程获取
- 相关性不确定 → 在 refined 笔记中列为"潜在相关来源"，标注为何未追踪
- 明显不相关 → 跳过，无需记录

---

## 3. 完整性规则

### 3.1 标签完整性

页面有多个标签（如"概述 | API | 示例 | 讨论"）且各标签内容不同时，分别收集。每个标签一个文件：

```
raw/ext-<name>-overview.md
raw/ext-<name>-api.md
raw/ext-<name>-examples.md
```

如果某标签内容与另一标签重复（如"打印视图"与"概述"相同），跳过并在文件中注释说明。

### 3.2 分页

对于分页列表（论坛、API 结果、"加载更多"按钮）：

- 默认上限：**10 页**（或第一个空页 / 列表结束信号，以先到者为准）
- 用户可在对话中指定更高的页数上限
- 停止条件（任一触发）：空页 / 与上一页内容相同 / 达到 max_pages / 404/410
- 每页保存为 `raw/ext-<name>-p{N}.md`，不合并多页

### 3.3 一级追踪

主页明确引用（"另见"、"相关文档"、"引用自"）且与意图锚定相关的外部文档，追踪一级。不递归。

- 追踪到的页面保存为 `raw/ext-<name>-ref-<short>.md`
- 不追踪装饰性链接（导航菜单、作者简介、页脚）

---

## 4. 元数据

每个 `raw/ext-*.md` 文件必须包含：

```yaml
---
url: <原始 URL>
tool: <使用的抓取工具>
fetched_at: YYYY-MM-DD HH:MM
---
```

分页文件额外包含 `page_of_set: <例如 "3/7">`。

---

## 5. 失败处理

URL 穷尽工具阶梯仍无法获取时：

1. **记录完整失败路径**到 `raw/ext-<name>-failed.md`，frontmatter 含 `url`、按顺序的 `attempts` 列表（每条 `tool` + `error`）、`final_status: unretrievable`。正文写"预期贡献"和可能的"替代方案 / 缺口确认"。
2. **披露到 `index.md`** 的"仍未解决的问题"或"是否适合进入 brainstorm"段：标 ⚠️ + 已尝试工具列表 + failed 文件路径。该输入仍关键时，状态不应标 `ready-for-brainstorm`。
3. **绝不捏造**：不从 URL、标题或搜索摘要推断不可获取来源的内容。

---

## 6. 边界 — 何时不使用本文件

本文件仅管辖 **Collect** 原语。不适用于：

- **Note** — 主题笔记是人工风格的整理与解释，不是重新抓取
- **Snapshot / Index** — `index.md` 只引用已收集的 raw 资料，不产生新抓取
- **Brainstorm** — brainstorm 不得抓取；如果 brainstorm 需要数据，应交回 research
- **Impl** — impl 读取 package PRD 的交付物/验收标准，不获取外部 URL
- **非 source 页面的 Wiki 摄取** — 仅 `source-*` 页面遵循此流程

---

## 7. 检查清单 — 声明收集完成之前

- [ ] 用户提供的 URL 全部已尝试；需登录的已请求用户协助
- [ ] 每次失败都按返回信号升级了工具，未静默放弃
- [ ] 多标签 / 分页 / 一级追踪 已完成（或确认不适用）
- [ ] 每个 `raw/ext-*.md` 都有 `url` / `tool` / `fetched_at` 元数据
- [ ] 失败的 URL 已写 `raw/ext-*-failed.md` 并在 `index.md` 披露
