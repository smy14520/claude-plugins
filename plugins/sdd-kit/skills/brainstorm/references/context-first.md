# Context first

存量项目中，代码阅读不是一次性动作，而是贯穿整个 brainstorm 的持续行为。

## 入口读取

已有代码、表结构、业务模块时,**第一轮追问前**先主动读取与需求相关的现有代码,把分析结论写入 PRD 的 `What I already know` 和 `Technical Framing`。

技术分析至少覆盖:

用户提供的 SQL / API / 接口契约视为基准约束，先原样沉淀到 PRD 或 artifact；与现有代码冲突时再用 `AskUserQuestion` 确认取舍。

- 涉及的现有表和关键字段。
- 现有模块 / 类的设计模式(适配器、策略、事件等)。
- 新需求的接入点:改哪些表 / 模块,建哪些新表 / 模块。
- 不能动的边界:其他模块依赖的接口、共享表等。

分析结论侧重**变更方案**(改什么、建什么、怎么接入),现有结构只需点明涉及的表 / 模块和关键关系,不需要完整复述。

## 持续读取

用户每次回答都可能让新的代码区域变得相关。展开新场景或用户提到新模块/能力时，先读取该区域的现有实现，再决定追问什么。从代码中发现的约束、模式和冲突是高价值问题的来源。

## 新项目

无既有代码 / 数据库时按常规工作循环进行,不需要前置代码分析。
