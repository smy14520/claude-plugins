## 初始用户需求

我想从零做一个轻量的个人 Todo 工具，帮助用户快速记录任务、按“今天/本周/以后”分组查看，并能用简单标签和完成状态管理日常待办。

## Brainstorm Turn 1

### 被测 agent

当前判断：这个需求的最大分叉不是“有哪些 Todo 功能”，而是首版到底是本地轻量工具，还是带账号/后端/同步的正式应用。

问题：首版你想把这个 Todo 工具做成哪种交付形态？

我推荐：**Local-first Web / PWA**。  
理由：单用户、网页可用、本地持久化、不做登录同步，最符合“轻量”和“从零快速交付”，可以先验证快速记录、分组、标签、完成状态这些核心体验。

可选方向：

1. **Local-first Web / PWA（推荐）**：本地保存数据，不需要登录和后端。
2. **账号同步 Web**：有登录、后端数据库、多端同步，更像正式 SaaS。
3. **移动优先 PWA**：仍可本地优先，但交互优先适配手机快速记录。
4. **CLI / 桌面工具**：偏键盘或桌面工作流，适合自用效率工具。

为什么现在问：它会决定后续的技术栈、数据持久化、是否需要登录权限、部署方式、测试策略和 PRD 的 slice 边界。

业务上我更关心首版能把“快速添加任务、按清单/分组整理、打标签、完成/恢复、搜索或筛选”这些日常体验打磨顺，不想把账号体系和同步复杂度带进第一版。

## Brainstorm Turn 2

### 被测 agent

当前判断：首版已经明确是 **Local-first 单用户 Web/PWA**，所以账号、后端、同步、团队协作都可以从 PRD 中剔除；现在最承重的产品决策变成“任务如何被归到今天/本周/以后”。

问题：**“今天 / 本周 / 以后”的分组依据，你希望首版怎么做？**

可选方向：

1. **手动时间桶 + 可选日期（推荐）**  
   任务主字段是 `bucket = today | week | later`；日期只是辅助信息。适合轻量、快速整理。
2. **纯 due date 自动归类**  
   用户填日期，系统按日期自动归到今天/本周/以后。更结构化，但快速添加成本更高。
3. **纯手动时间桶**  
   只有今天/本周/以后，没有日期。最简单，但以后可能难以表达“周三做”或“下月某天”。
4. **清单优先，时间桶只是筛选**  
   任务主要归到清单，今天/本周/以后只是基于任务属性的视图。更灵活，但首版模型会更复杂。

为什么现在问：它会直接决定 Task 数据模型、添加任务表单、分组视图规则、搜索/筛选条件和验收标准。

选 **手动时间桶 + 可选日期**。

日期可以作为可选辅助：比如我把任务放到“本周”，同时标一个“周三”；或者放到“以后”，标一个下月某天。这样用户想细化时有空间，但不填日期也不影响任务正常管理。

业务规则上我倾向于：

- `bucket` 是主要分组依据，用户手动选择。
- 日期不自动改变任务所属 bucket。
- 如果任务有日期，可以在卡片上显示出来，并支持按日期筛选或搜索。
- 任务从“本周”拖到“今天”时，就是用户主动调整优先级，不需要系统根据日期纠正。
