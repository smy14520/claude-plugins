# Brainstorming — 将想法打磨为设计方案

通过自然的协作对话，将模糊的想法转化为完整的设计方案。先理解项目上下文和历史经验，再逐步提问，最终输出经过验证的设计文档。

<USAGE-GATE>
仅在以下情况进入 brainstorming 流程：
- 用户明确要求使用 brainstorming / 设计流程
- 需求存在明显歧义，需要先澄清再实现
- 涉及跨模块改动、行为变更较大、或确实需要 2-3 个方案对比

如果需求很小、范围明确、改动集中，可以直接给出简短方案或直接实现，不必进入本流程。
一旦进入本流程，在完成设计方案并获得用户明确确认之前，不得调用任何实现类 Skill、不得写任何代码、不得搭建任何项目骨架、不得执行任何实现操作。
</USAGE-GATE>

## 何时不该用 brainstorming

以下情况通常**不需要**进入本流程：
- 单文件或小范围修改
- 明确的 bug fix
- 用户已经给出足够具体的实现方向
- 仅需小的命名、文案、样式或配置调整

这类请求优先直接分析、直接提议、直接修改。

---

## Checklist

进入本流程后，按顺序完成以下步骤：

1. **探索项目上下文** — 查文件、文档、近期 commits；只有在需要知识库辅助时，才调用 ContextDetective 检索经验/风险
2. **评估需求范围** — 判断是否需要分解为子项目
3. **提供 Visual Companion**（如涉及视觉问题）— 这是独立的一条消息，不与其他问题混合
4. **逐一提问澄清** — 每次只问一个问题，理解目的/约束/成功标准
5. **提出 2-3 个方案** — 对比优劣，给出推荐
6. **分段呈现设计** — 按复杂度缩放每段长度，逐段确认
7. **写入设计文档** — 保存到 `.claude/plans/<需求名>-plan.md` 并 git commit
8. **Spec 自检** — 4 项内检（占位符/一致性/范围/歧义）
9. **用户审阅门禁** — 请用户审阅文件，确认后才可进入下一步
10. **过渡到任务规划** — 衔接 `/tasks` 命令

## 详细流程

### Step 1: 探索项目上下文（知识优先）

**在问任何问题之前，先了解项目现状和历史经验。**

1. **先查看项目文件** — 检查目录结构、配置文件、近期 commits
2. **按需调用 ContextDetective** — 仅当需求涉及已有知识库、风险规则、模块边界不清，或用户明确要求检索上下文时，才获取风险规则、相关经验、Gotchas、模块索引
3. **检查已有设计** — 看 `.claude/plans/` 下是否有相关的历史设计方案

简要输出：
```
📋 项目上下文
- 技术栈: [从项目配置识别]
- 相关经验: [来自 ContextDetective，列出匹配的经验文档]
- 风险提示: [来自 ContextDetective，列出匹配的风险规则]
- 历史设计: [如有相关的 plans]
```

### Step 2: 范围评估

在深入设计前先判断范围：

- 如果需求描述了**多个独立子系统**（如"做一个包含聊天、文件存储、计费和分析的平台"），立即指出
- 不要花时间细化一个需要先分解的大项目
- 帮助用户分解为子项目：有哪些独立部分？它们如何关联？按什么顺序构建？
- 然后对**第一个子项目**走正常的设计流程
- 每个子项目独立走 design → tasks → impl → extract-experience 循环

### Step 3: Visual Companion（可选）

当你预判接下来的问题会涉及**视觉内容**（mockup、布局、图表）时，单独发一条消息提供：

> "接下来的一些讨论用图来解释可能更直观。我可以在浏览器里展示 mockup、布局对比、架构图等可视化内容。这个功能需要打开一个本地 URL。要试试吗？"

**这条消息必须独立发送**，不要和其他问题或内容混在一起。等用户回复后再继续。如果用户拒绝，用纯文本方式继续。

如果用户同意，阅读详细指南：`skills/brainstorming/visual-companion.md`

**逐题判断**：即使用户同意了 companion，也要对每个问题单独判断——这个问题用浏览器还是终端更好？

- **用浏览器**：UI mockup、布局对比、架构图、视觉设计
- **用终端**：需求问题、概念选择、优劣对比、技术决策

### Step 4: 逐一提问澄清

- 每条消息**只问一个问题**
- **优先选择题**，比开放题更容易回答
- 聚焦理解：目的、约束、成功标准
- 如果某个主题需要深入，拆成多个问题

问题聚焦方向（按需选用，不要全部问一遍）：

| 层面 | 关注点 |
|------|--------|
| **产品需求** | 目标与价值、用户与场景、MVP 范围、成功标准 |
| **架构决策** | 仓库模式、部署方式、渲染模式、认证与权限 |
| **技术选型** | 前后端技术栈、数据库、API 风格、测试策略 |

**访谈结束标志**：已清楚做什么、为谁做、怎么判断成功，或用户说"可以开始设计了"。

### Step 5: 提出 2-3 个方案

- **必须**提出 2-3 个方案进行对比
- 领先推荐方案，说明为什么
- 标注各方案优缺点

```
## 方案 A: [名称]（⭐ 推荐）
- 优点: ...
- 缺点: ...

## 方案 B: [名称]
- 优点: ...
- 缺点: ...

我推荐方案 A，因为 [理由]。你倾向哪个？
```

### Step 6: 分段呈现设计

- 按复杂度缩放每段：简单的几句话，复杂的 200-300 字
- 每段之后问用户是否满意
- 覆盖：架构、组件、数据流、错误处理、测试
- 随时可以回溯修改

**设计原则**：
- 拆分为职责单一、接口清晰、可独立理解和测试的小单元
- 每个单元能回答：它做什么？怎么用？依赖什么？
- YAGNI — 在输出最终设计前，主动列出可以移除或推迟的功能

**现有代码库中**：
- 先了解现有结构，遵循现有模式
- 如果现有代码影响了当前工作（文件过大、职责不清），在设计中包含针对性改进
- 不做无关的重构

### Step 7: 写入设计文档

保存到 `.claude/plans/<需求名>-plan.md`：

```markdown
---
name: <需求名>
project: <项目名>
created: <日期>
architect: brainstorming
context_detective_report: <是否使用了 ContextDetective>
related_experience: [<匹配到的经验文档列表>]
risk_warnings: [<匹配到的风险规则列表>]
---

# <需求名> 设计方案

## 1. 背景与目标
### 1.1 问题描述
### 1.2 目标用户
### 1.3 成功标准

## 2. 项目上下文
### 2.1 相关经验（来自知识库）
### 2.2 风险提示（来自规则库）

## 3. 方案选择
### 3.1 备选方案对比
### 3.2 最终选择及理由

## 4. 技术设计
### 4.1 整体架构
### 4.2 组件职责
### 4.3 数据流
### 4.4 API 设计

## 5. 边界与约束
### 5.1 边界条件
### 5.2 错误处理
### 5.3 非功能需求

## 6. 风险与建议
### 6.1 潜在风险
### 6.2 后续优化建议
```

**写完后执行 git commit**：
```bash
git add .claude/plans/<需求名>-plan.md
git commit -m "design: <需求名> 设计方案"
```

### Step 8: Spec 自检

写完设计文档后，用全新的眼光审视它：

| 检查项 | 内容 |
|--------|------|
| **占位符扫描** | 有没有 TBD、TODO、不完整的部分？修掉 |
| **内部一致性** | 各章节之间有没有矛盾？架构是否匹配功能描述？ |
| **范围检查** | 是否聚焦到一个可实现的单元？还是需要继续分解？ |
| **歧义检查** | 有没有需求可以被解读为两种意思？如果有，选一种并写明 |

发现问题就直接修掉，不需要重新 review。

### Step 9: 用户审阅门禁

> "设计方案已写入并提交到 `.claude/plans/<需求名>-plan.md`。请审阅后告诉我是否需要修改，确认后我们开始任务规划。"

**等待用户回复。** 如果用户要求修改，修改后重新自检。只有用户确认后才可进入下一步。

### Step 10: 过渡到任务规划

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 设计方案已完成

文件: .claude/plans/<需求名>-plan.md
相关经验: <列出匹配的经验文档>
风险提示: <列出匹配的风险规则>

下一步: /tasks <需求名>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 核心原则

- **知识按需使用** — 需要知识库辅助时再检索经验和风险，不为流程而检索
- **每次只问一个问题** — 不要用问题轰炸用户
- **选择题优先** — 比开放题更容易回答
- **YAGNI 无情** — 删除所有不必要的功能
- **探索多种方案** — 需要设计时提出 2-3 个方案再选择
- **渐进式验证** — 分段呈现，逐段确认
- **设计门禁只在进入流程后生效** — 不要把 brainstorming 误用为所有小改动的前置步骤
